Why I added backpressure and load shedding

A proxy that accepts every request during a spike does not stay up longer. It fails slower, and for everyone at once. Here is why an unbounded queue is the real failure mode, and why a fast 429 is a kindness.

The decision

Cap concurrency, queue a little, shed the rest

The default that betrays you

  • Accept every connection; let an unbounded queue absorb overflow.
  • Backlog grows, so memory grows with it.
  • Every caller's latency rises together, even cheap requests.
  • Timeouts fire, clients retry, retries add load: the retry storm.
  • It looks like resilience and is the opposite.

The admission gate

  • Process at most MAX_INFLIGHT requests at once.
  • Let a small bounded number wait briefly for a slot (MAX_QUEUE).
  • Shed the rest now with 429 and a Retry-After header.
  • Admitted requests run at full speed; the tail stays bounded.
  • A fast rejection returns control to the client while it still has options.
The subtlety

Why limit concurrency, not a requests-per-second rate

A fixed requests-per-second cap is brittle, because the cost of a request varies. Limiting concurrent in flight work adapts on its own: when the upstream slows down, in flight requests linger, fewer slots free up, and the gate starts shedding sooner without anyone re-tuning a number. Concurrency is the control variable that maps to real resource pressure.

This is also why the admission wait is counted inside an admitted request's latency. The time a request spends queued is latency the client really felt, so the observatory's end-to-end numbers stay honest. A shed request, by contrast, is not a served request, so it is kept out of the success and latency stats and counted in its own rejected bucket.

Under the hood

One lock, no leaked slots

The tempting implementation, a semaphore plus a side counter, has a cancellation race: a timeout can fire just after the permit is handed out, leaking a slot forever. So the limiter keeps all of its state behind a single asyncio.Condition; every transition is one atomic step.

acquire():
  under the condition lock:
    if closed:                       -> Rejected("shutdown")
    if a slot is free AND nobody waits ahead:
        active += 1; return          # fast path
    if waiting >= max_queue:          -> Rejected("queue_full")
    waiting += 1
    loop until the deadline:
        if closed:                   -> Rejected("shutdown")
        if a slot is free:           active += 1; return
        wait on the condition for the remaining budget
        if the budget is gone:       -> Rejected("queue_timeout")
    finally: waiting -= 1, and re-notify the next waiter if we
             were woken but did not take the slot

The fairness gate (nobody waits ahead) stops a steady arrival stream from starving a parked waiter. The finally re-notify makes sure a wake-up is never swallowed, so a free slot is never stranded.

The knobs

Opt-in, off by default

EnvDefaultMeaning
MAX_INFLIGHT0Max requests processed at once. 0 disables the gate and the request path is unchanged.
MAX_QUEUE0How many may wait for a slot. 0 is pure fail fast.
ACQUIRE_TIMEOUT_S0.5Longest a queued request waits before it is shed.
RETRY_AFTER_S1Value sent in the Retry-After header on a 429.

With METRICS_ENABLED=1, GET /metrics grows a rejected block (total plus per-reason counts) and a live concurrency gauge, so total inbound is requests_total + rejected_total.
